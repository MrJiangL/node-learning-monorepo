# MySQL 集成测试隔离复盘

## 为什么集成测试更容易互相影响？

集成测试会读写同一个真实数据库。

如果多个测试共用同一套 MySQL 数据库，那么一个测试创建的数据，可能会被另一个测试读到；一个测试清理数据库，也可能把另一个测试刚创建的数据删掉。

所以集成测试比纯函数单元测试更容易互相影响。单元测试通常只依赖函数输入和内存里的 fake 数据，而集成测试会依赖数据库这种共享外部状态。

## 为什么要清理数据库？

不清理数据库会导致后面的测试受到前面测试数据的污染。

例如一个测试创建了 3 个 Project，如果下一个测试想验证“当前用户没有 Project”，但数据库里还残留旧数据，这个测试就会失败，或者更糟糕：因为读到了不该读的数据，测试结果变得不准确。

所以每个集成测试开始前，都应该让数据库回到一个可预期的干净状态。

## 为什么清理顺序要从子表到父表？

因为数据库里有外键关系。

例如：

```text
Todo -> Project -> User
Plan -> User
UserSession -> User
```

`Todo` 依赖 `Project`，`Project` 依赖 `User`。如果先删除父表 `User`，但还有 `Project`、`Plan`、`UserSession` 指向这个 User，数据库会阻止删除，避免留下无效的外键引用。

所以清理顺序要从“依赖别人”的子表开始：

```text
Todo -> Project / Plan / UserSession -> User
```

这样每一步删除时，都不会破坏外键约束。

## 为什么 API 测试要 `fileParallelism: false`？

`fileParallelism: false` 是为了让不同测试文件不要同时访问同一个 MySQL 测试库。

如果多个测试文件并行执行，就可能出现这种情况：

```text
plans.test.ts 创建了一个 Plan
projects.test.ts 的 beforeEach 正在清理 User
plans.test.ts 继续查询刚才的 Plan
结果数据已经被别的测试文件清掉了
```

这类问题不是业务代码错，而是测试文件并发访问同一个数据库导致的测试隔离问题。

学习阶段先让测试文件串行执行，可以牺牲一点速度，换来更稳定、更容易理解的测试结果。

## 以后如果测试变慢，可以怎么优化？

1. 给每个测试文件独立数据库
2. 每个测试用 transaction，测试结束 rollback
3. 只让真正访问数据库的测试串行，纯单元测试继续并行
4. 减少不必要的集成测试，把纯业务逻辑留给 service 单元测试
