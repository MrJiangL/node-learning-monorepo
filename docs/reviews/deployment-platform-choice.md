# 部署方式选择

## 1. 我当前项目要部署哪些东西

当前项目不是只有一个 Node 服务，而是一个小型 monorepo。

需要部署或准备的东西可以拆成几块：

| 部分              | 当前内容                         | 第一版是否必须部署       |
| ----------------- | -------------------------------- | ------------------------ |
| API 服务          | `apps/api`，Express + TypeScript | 必须                     |
| Web 前端          | `apps/web`，Vite + Vue           | 可以后置                 |
| MySQL             | Prisma 连接的业务数据库          | 必须                     |
| Redis             | Project 列表缓存                 | 可以后置或使用托管 Redis |
| Job Worker        | 数据库队列后台任务               | 第一版建议先关闭         |
| OpenAPI / Swagger | API 文档页                       | 跟随 API 一起部署        |
| GitHub Actions    | CI 自动验证                      | 已完成                   |

第一版部署的目标不应该是“一次性把所有东西都生产化”。

更合理的目标是：

1. 先让 API 在线跑起来。
2. 能连上生产数据库。
3. 能执行 Prisma migration。
4. 能通过健康检查接口确认服务活着。
5. 再逐步补 Redis、Worker、Web 前端和监控。

## 2. 方案 A：托管平台

托管平台指 Render、Railway、Fly.io、Vercel、Netlify 这类服务。

它们会帮我处理很多服务器运维细节，例如：

- 代码拉取
- 构建命令
- 启动命令
- 环境变量配置
- 日志查看
- 自动重启
- HTTPS
- 部分平台还提供数据库或 Redis

我需要重点关心的是：

- Build command 怎么写
- Start command 怎么写
- 环境变量填哪些
- 数据库连接字符串怎么配置
- `prisma migrate deploy` 在哪里执行
- 服务启动后怎么通过 `/health` 检查

优点：

- 上手快
- 不需要先学 Linux 运维
- 更适合第一次部署
- 能专注理解 Node 项目上线链路
- 和 GitHub 连接后可以自动部署

缺点：

- 免费额度有限
- 平台有自己的规则
- 后台 worker、多服务拆分时需要多理解平台模型
- 有些平台不一定直接提供 MySQL，可能要接外部数据库

对我当前阶段来说，托管平台更合适。

## 3. 方案 B：自管云服务器

自管云服务器指自己买一台 VPS / 云服务器，然后自己配置运行环境。

例如：

- 阿里云 ECS
- 腾讯云 CVM
- AWS EC2
- DigitalOcean Droplet
- Hetzner VPS

如果走这条路线，我要自己处理：

- Linux 用户和权限
- Node.js 安装
- npm / pnpm 安装
- MySQL 安装或远程连接
- Redis 安装或远程连接
- Nginx 反向代理
- HTTPS 证书
- PM2 或 Docker
- 防火墙
- 服务器日志
- 服务崩溃自动重启
- 系统安全更新

优点：

- 控制力强
- 更接近传统生产环境
- 能完整学习服务器运维
- 后续可以自己部署多个服务

缺点：

- 第一次部署成本高
- 问题面太宽，容易卡在 Linux / Nginx / 证书上
- 会分散 Node 后端学习重点
- 安全责任更重

所以我现在不应该第一步就买服务器。

自管服务器可以作为后续阶段：当我已经在托管平台跑通 API、数据库、环境变量、migration 和日志之后，再回来学 VPS / Docker / Nginx，会更稳。

## 4. 方案 C：混合部署

混合部署是把不同部分放到不同平台：

- Web 前端放 Vercel / Netlify
- API 放 Render / Railway / Fly.io
- MySQL 用托管数据库
- Redis 用托管 Redis
- Worker 单独作为一个后台服务部署

这更接近现代小团队常见方式。

优点：

- 每个服务用最适合的平台
- 前端部署体验好
- API 和 worker 可以独立扩展
- 数据库和 Redis 由专门平台托管

缺点：

- 环境变量分散在多个平台
- 跨域、API URL、CORS 要处理
- 日志分散
- 新手第一次会有点乱

对当前项目来说，混合部署可以作为第二阶段。

第一阶段先只部署 API，等 API 跑通后，再部署 Web 前端并配置 API base URL。

## 5. 我现在是否需要先准备服务器

我现在不需要先准备自己的服务器。

原因是：

1. 我是第一次部署，最重要的是先理解部署链路，而不是马上学习所有运维细节。
2. 当前项目已经有 CI、环境变量、Prisma、MySQL、Redis、worker 等很多概念，如果再加入 Linux / Nginx / SSL，复杂度会突然变高。
3. 托管平台已经能覆盖第一版学习目标：构建、启动、环境变量、数据库连接、日志、健康检查。
4. 自管服务器不是不用学，而是应该放在后面学。

所以当前选择是：

先不用买服务器。

先选择一个托管平台，把 API 服务部署成功。

## 6. 我的第一版部署选择

第一版部署建议：

| 项目部分 | 第一版选择                 | 理由                                    |
| -------- | -------------------------- | --------------------------------------- |
| API      | Render 或 Railway          | 上手快，支持 GitHub 自动部署            |
| MySQL    | 托管 MySQL / 云数据库      | 避免自己维护数据库进程                  |
| Redis    | 暂时关闭或托管 Redis       | 当前缓存有 fallback，不必阻塞第一版部署 |
| Worker   | `JOB_WORKER_ENABLED=false` | 避免第一版部署时引入后台轮询复杂度      |
| Web      | 后置部署                   | 先把 API 跑通，再接前端                 |

第一版成功标准：

1. 平台能从 GitHub 拉代码。
2. 能安装依赖。
3. 能执行 API build。
4. 能配置生产环境变量。
5. 能连接生产 MySQL。
6. 能执行 `prisma migrate deploy`。
7. API 能启动。
8. `/health` 能访问。
9. Swagger / API docs 能访问。

第一版部署完成后，再进入下一步：

- 加 Redis
- 拆 worker
- 部署 Web
- 配 CORS
- 配生产日志
- 配监控和告警

## 7. 我以后怎么做这种复盘

我现在理解，复盘不是写标准答案，也不是证明自己都懂了。

复盘更像是把一团新知识整理成几个问题：

1. 我这次面对的是什么问题？
2. 它有哪些可选方案？
3. 每个方案的代价是什么？
4. 以我当前阶段，哪个方案最适合？
5. 下一步最小行动是什么？

以后遇到不会的工程问题，可以先按这个模板写：

### 1. 这件事要解决什么

用自己的话写清楚目标。

例如：部署不是“把代码传上去”，而是让服务在一台远程机器或平台上稳定运行。

### 2. 我有哪些选择

列出 2 到 4 个方案。

例如：托管平台、自管服务器、混合部署。

### 3. 每个选择的好处和坑

不要只写优点，也要写缺点。

例如：自管服务器控制力强，但第一次部署容易被 Linux / Nginx / SSL 卡住。

### 4. 我当前阶段选哪个

结合自己的学习目标做选择。

例如：我现在先选托管平台，因为我要先学 Node 项目上线链路。

### 5. 下一步最小动作

把下一步压小。

例如：不是“完成所有部署”，而是“先选择平台，并整理 API 的 build/start/env 配置”。

这个复盘方式的重点是：先降低混乱度，再进入实践。

我不需要第一次就把所有部署知识学完。先跑通一条最小链路，再逐步加复杂度。
