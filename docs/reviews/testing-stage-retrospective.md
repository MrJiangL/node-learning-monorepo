# 测试强化阶段复盘

## 1. Service 测试在测什么

Service 测试主要测业务规则和协作者调用有没有正确发生。

比如创建、更新、删除 Project / Todo 时，service 应该调用 repository，也应该调用 ActivityLogService 记录日志。

它不重点测试 Prisma 怎么查数据库，而是测试“业务流程有没有走对”。

## 2. Repository 测试在测什么

Repository 测试主要测真实数据库读写和查询条件有没有生效。

比如 `action` 过滤、`createdAfter` / `createdBefore` 时间范围、分页和排序，这些都属于 repository 更适合保护的范围。

它关心的是：Prisma 查询条件写出来以后，数据库返回的数据是不是符合预期。

## 3. API 集成测试在测什么

API 集成测试主要测 HTTP 入口到 response 出口的完整链路。

比如未登录返回 401、非法 query 返回 400、成功请求返回 `success/data/meta`，这些都是 API 对外契约。

它可以经过 middleware、Zod validation、service、repository，但重点不是验证每一层内部细节，而是验证调用接口的人看到的结果。

## 4. 什么情况下测试是必须写的

我觉得关键业务规则、权限边界、数据写入、复杂查询、错误响应契约，最好写测试。

比如“用户不能看别人的 Project”、“Project 删除后仍然能查到删除日志”、“非法日期 query 要返回校验错误”，这些如果坏了影响比较大。

这种测试不是为了追求数量，而是为了保护最容易出事故的地方。

## 5. 什么情况下可以先不写测试

特别简单的展示逻辑、很薄的透传代码、临时学习代码，可以先不写测试。

如果一段代码没有复杂分支，也不涉及权限、数据库写入、金额、状态流转，就可以先靠类型检查和人工验证。

但是后面一旦它变成核心流程，或者出现过 bug，就应该补测试。

## 6. 我现在写测试最卡的地方

我现在最卡的是测试设计：知道测试能保护代码，但不太确定一个需求应该放在 Service、Repository，还是 API 集成测试里。

数据准备也比较卡，因为有些测试要先创建 user、project、todo、activity log，链路一长就容易不知道从哪里开始。

目前比较适合的练法是：先看懂现有测试，再改一个条件，最后再自己设计一个新 case。
