# 幂等入门：idempotency key

## 1. 什么是幂等

幂等是指同一个业务请求执行一次和多次，最终业务结果应该一致。

这里的“结果一致”不是说接口一定返回完全一样的时间戳或响应细节，而是说业务上不能重复创建两份数据。

## 2. 什么场景会出现重复请求

重复请求 可能来自用户 重复点击 前端超时重试 队列重复投递 服务间调用重试

更具体地说：

```text
1. 用户连续点击提交按钮。
2. 前端请求超时后自动重试。
3. 服务 A 调服务 B，没收到响应，于是重试。
4. 队列系统至少投递一次，同一个消息可能被 worker 收到两次。
```

## 3. idempotency key 解决什么问题

idempotency key 是客户端或者调用方为一次业务操作生成的唯一标识。

后端收到重复的 key 时，应该识别这是同一次业务操作，而不是创建新的业务数据。

比如同一个创建 Job 请求带着同一个 `idempotencyKey`：

```text
第一次请求：创建 Job A。
第二次请求：发现这个 key 已经创建过 Job A，于是返回 Job A，不再创建 Job B。
```

## 4. 它和 transaction / lock 的区别

transaction 解决 一组数据库操作要么一起成功，要么一起失败。

lock 解决多个并发流程同时修改同一份数据的竞争。

idempotency key 解决同一个业务请求被重复执行的问题。

它们解决的问题不一样：

```text
transaction：防止半成功半失败。
lock：防止多个流程同时抢同一份数据。
idempotency key：防止同一个业务请求被重复执行后创建重复数据。
```

## 5. 我理解中的实现思路

我的理解是：

```text
1. 请求里带一个 idempotencyKey。
2. 后端先查这个 key 是否已经创建过业务数据。
3. 如果已经存在，就返回已有数据。
4. 如果不存在，就创建新数据，并把 idempotencyKey 一起保存。
5. 数据库最好对 idempotencyKey 做唯一约束，避免并发重复创建。
```

所以幂等不只是 if 判断，最好要有数据库唯一约束兜底。
